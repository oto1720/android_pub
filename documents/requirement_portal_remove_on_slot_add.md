# 要件: スロット追加時に Portal から記事を除外する

## 概要

Portal 画面で「スロットに追加」ボタンを押した記事は、積読スロットに追加された後、Portal の記事一覧から即座に消えるべきである。

---

## 背景・現状

- 現在、スロット追加後も該当記事は Portal 一覧に残ったまま
- DB 上では `status` が `PORTAL` → `TSUNDOKU` に更新されるが、Portal の UI は初回ロード時のスナップショットを保持しており、リアルタイムに反映されていない

---

## 機能要件

### FR-1: 一覧からの即時除外

| 項目 | 内容 |
|------|------|
| **トリガー** | ユーザーが Portal の記事カードで「スロットに追加」ボタンを押下し、追加が成功したとき |
| **期待動作** | その記事が Portal の記事一覧から即座に消える |
| **対象** | 追加に成功した記事のみ（スロット満杯で失敗した場合は一覧に残る） |

### FR-2: 除外のタイミング

| 項目 | 内容 |
|------|------|
| **タイミング** | 追加処理が成功した直後（DB 更新完了後） |
| **遅延** | ユーザーが体感できる遅延なく、自然に消えること |

### FR-3: 他画面との整合性

| 項目 | 内容 |
|------|------|
| **積読画面** | 追加した記事は Tsundoku 画面のスロットに表示される |
| **空きスロット表示** | Portal ヘッダーの「空きスロット X / 5」が正しく減る（既存仕様どおり） |

---

## 非機能要件

### NFR-1: 一覧の並び順

- 記事を除外した後も、残り記事の表示順は変わらない
- 新たな API 呼び出しや一覧の再ソートは不要

### NFR-2: エラー時

- スロット満杯で追加失敗した場合: 一覧は変更しない（既存の Snackbar 通知のみ）
- DB エラー等で追加失敗した場合: 一覧は変更しない（将来の例外ハンドリングで対応）

### NFR-3: タグフィルタ時

- タグでフィルタしている場合も、除外の挙動は同じ
- 除外後、一覧が空になった場合は既存の空状態 UI を表示

---

## 完了条件

- [ ] Portal でスロット追加に成功した記事が、一覧から即座に消える
- [ ] スロット満杯で失敗した場合は一覧に残り、Snackbar が表示される
- [ ] タグフィルタ有無にかかわらず同じ挙動である
- [ ] 既存の `addToTsundoku` の単体テストが通る
- [ ] 本要件を満たすための変更に対するテストが追加されている

---

## 備考

- 実装方針の候補:
  - **A) 楽観的 UI 更新**: `addToTsundoku` 成功時に、ViewModel の `uiState.articles` から該当 `articleId` を除外する
  - **B) リアクティブ監視**: Portal の記事一覧を `observePortalArticles()` の Flow で監視し、status 変更を自動反映する（アーキテクチャ変更が大きい）
- 現状の `observePortalArticles()` は `status = PORTAL` のみを返すため、DB 上では追加済み記事は既に除外されている。問題は Portal が一度きりのロード結果を保持している点である。
