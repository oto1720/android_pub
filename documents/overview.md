# Tech-Digest プロダクト概要

## プロダクト名

**Tech-Digest**

## 一言で言うと

> 「読んだつもり」をなくす、技術記事の **理解完了アプリ**

---

## 解決する課題

若手開発者・プログラミング学生が日常的に抱える3つの問題を解決する。

| # | 課題 | 具体的な状態 |
|---|------|------------|
| ① | **「読んだつもり」問題** | Qiita・Zennを読んでも実装時に思い出せず、知識が定着しない |
| ② | **積読の心理的負債** | ブックマークが際限なく増え、未消化リストがストレスになる（技術FOMO） |
| ③ | **チュートリアル地獄** | 記事Aを読んで疑問が生まれ、記事B・Cへ…と迷子になり疲弊する |

---

## ターゲットユーザー

- プログラミングサークルの大学生
- 開発歴 1〜2年目の若手エンジニア
- 通学中・就寝前にスマホで技術記事を読み漁る習慣がある人

---

## ソリューション（コアコンセプト）

### 「情報を制限し、消化を強制するワークフロー」

| 課題 | 解決策 |
|------|--------|
| ① 知識未定着 | **強制1行メモ（AI壁打ち）** — 記事を読んだだけでは完了にならない。AIが**記事全文**を読んで内容に即した理解度チェック問いを出し、1行メモを書いてAIが「理解十分」と判定してはじめて「消化」できる |
| ② 積読膨張 | **積読スロット制限（最大5件）** — 上限に達すると新規追加ができなくなり「まず消化してから次へ」という健全なサイクルを強制する |
| ③ 迷子防止 | **オフライン集中モード** — 記事をローカル保存し、ネット接続を遮断した状態で目の前の1記事だけに集中できる |

---

## 主な画面と機能

```
ポータル画面（トレンド一覧）
 └─ Qiita APIからトレンド記事を取得して表示
 └─ タイトル・冒頭文（プレビュー）のみ表示（全文は読めない）
 └─ タグ別絞り込み
 └─ 「スロットに追加」ボタン（スロット満杯時はロック）
 └─ ※記事タップで詳細表示はしない（スロット追加後に積読画面で読む）

積読リスト画面（最大5枠）
 └─ AI要約ボタン（Gemini APIが記事全文を読んで要約を生成）
 └─ 記事本文を読んで理解度チェックへ進む

消化フロー（理解度チェック）
 └─ AIが記事全文を読んで内容に即した問いを生成
 └─ ユーザーが1行メモを入力・送信
 └─ AIがフィードバックを返す ＋ 理解度を評価（十分/不十分）
 └─ 理解十分 → 「消化する」ボタン有効化 → 血肉リストへ
 └─ 理解不十分 → 「消化する」ボタン無効、再回答を促す

血肉リスト画面（消化済み一覧）
 └─ 消化済み記事・メモ・AIフィードバックを確認
```

## ユーザー体験の変化

| Before | After |
|--------|-------|
| 大量に記事を読んだのに何も作れていない… | この記事の概念を1つ完全に理解した！ |
| 焦燥感・自己肯定感の低下 | 小さな成功体験・確かな前進 |

## 技術スタック

| カテゴリ | 採用技術 |
|---------|---------|
| プラットフォーム | Android (Kotlin) |
| UI | Jetpack Compose |
| アーキテクチャ | Clean Architecture + MVVM |
| 状態管理 | StateFlow / UiState |
| DI | Hilt |
| ローカルDB | Room |
| 外部API | Qiita API（記事取得）、Gemini API（AI機能） |
| ネットワーク | Retrofit |

---

## アーキテクチャ概要

```
UI層（Compose）
  ↓
ViewModel層（StateFlow）
  ↓
UseCase層（Clean Architecture）
  ↓
Repository層
  ↓
DataSource層（Qiita API / Gemini API / Room DB）
```
