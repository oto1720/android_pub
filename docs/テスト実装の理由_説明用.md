# なぜテストを実装したのか — 説明用

> 発表・面接で「テストをどう考えて実装したか」を説明できるようにするためのドキュメントです。  
> 30秒〜1分で話せる型と、突っ込み質問への回答をまとめています。

---

## 1. 説明の型（話すときのフォーマット）

### 1分で説明する型

```
【目的】
テストの目的は「コードが動く証拠」ではなく、「壊れたときに即座に気づける仕組み」を作ることです。

【方針】
このプロジェクトでは、壊れたら UX やデータ整合性が崩壊する箇所を重点的にテストしています。
E2E は書かず、単体テストと DAO の統合テストに絞りました。

【具体例】
例えば積読の5件制限。ここが壊れるとコア仕様が崩れるので、SQL レベルで検証しています。
API の 429 エラーを適切なメッセージに変換するロジックも、壊れるとユーザーに伝わらなくなるのでテストしています。

【書かないもの】
定数のテストや「1回呼ばれた」だけのテストは、コストに対して得られる保証が小さいと判断して書いていません。
```

### 30秒で説明する型

```
「壊れたら困るところだけを重点的にテストしています。
積読5件制限、ユーザーデータの上書き保護、エラーのマッピング、外部キーの CASCADE の4つです。
Mock で依存を制御し、DAO だけは実際の SQLite で動かして SQL のロジックを検証しています。」
```

## 2. なぜテストを書くのか（4つの効果）

| 効果 | 一言で | 説明用 |
|------|--------|--------|
| **回帰防止** | 壊れたらすぐ気づく | リファクタリングや機能追加で既存の挙動を壊したとき、CI が即座に検知する |
| **設計の圧力** | テストしやすさが設計の指標 | テストが書きにくいコードは、依存が多すぎる・責務が大きすぎるサイン |
| **ドキュメント代替** | テスト名が仕様書 | `updateStatusIfSlotAvailable - スロットが満杯のとき - 0を返す` のように、何を守っているかが名前でわかる |
| **安全なリファクタリング** | 変更を恐れない | テストが通る限り、内部実装を自由に変えられる |

---

## 3. 何をテストしているか（4つの重点）

| テスト | 何を守る | 壊れたらどうなる |
|--------|----------|-------------------|
| **スロット制限**（ArticleDaoTest） | 積読5件まで | 「5件まで」というコア仕様が崩壊する |
| **上書き保護**（ArticleRepositoryImplTest） | 積読・消化済みが消えない | ユーザーの積読リストが API リフレッシュのたびに消える |
| **エラーマッピング**（ArticleRepositoryImplTest） | 429 → RateLimitExceeded | レート制限時に適切なメッセージを表示できない |
| **外部キー CASCADE**（DigestDaoTest） | 記事削除で消化記録も削除 | ゴースト消化記録が DB に残り続ける |

## 4. なぜその方法でテストするか

### なぜ Mock を使うか

| 理由 | 説明 |
|------|------|
| 依存の制御 | 「API が 429 を返したとき」を再現して、Repository のエラーマッピングだけを検証したい |
| 実際の API を呼ばない | ネットワーク依存、レート制限、CI の不安定さ、テストの遅さを避ける |
| テスト対象の孤立 | 特定のロジックだけを検証できる |

**一言**: 「依存を制御して、テスト対象だけを検証するためです。」

### なぜ DAO は InMemory Room か

| 理由 | 説明 |
|------|------|
| SQL は Mock では検証できない | `updateStatusIfSlotAvailable` の WHERE 句が本当にスロット満杯のとき更新しないかは、実際の SQLite に流して確認する必要がある |
| 実際の DB で動く | InMemory Room は実際の SQLite。テスト終了後はメモリから消える |
| ネットワーク不要 | ローカルで実行できる |

**一言**: 「SQL のロジックは Mock では検証できないので、実際の SQLite で動かしています。」

### なぜ E2E を書かないか

| 理由 | 説明 |
|------|------|
| コストが高い | 実装・メンテナンスに時間がかかる |
| 壊れやすい | UI のちょっとした変更で落ちる |
| CI が遅くなる | エミュレータ起動や画面操作で時間がかかる |
| 代替で十分 | 壊れたら困る箇所を単体・統合テストでカバーしている |

**一言**: 「コストが高く壊れやすいので、単体・統合テストで重点箇所をカバーする方針にしています。」

---

## 5. 書かないテストとその理由

| 書かないテスト | 理由 |
|----------------|------|
| 定数テスト（`assertEquals(5, MAX_SLOTS)`） | 定数が変わったらテストも直すだけ。壊れても UX に影響しない |
| 「1回だけ呼ぶ」テスト | 正常系が通れば「呼ばれた」は自明 |
| Retrofit アノテーション確認 | フレームワークの動作保証。Retrofit が壊れたら他も全滅 |
| data class のフィールドチェック | フィールドを変えたらコンパイルエラーで気づく |
| 実 API を叩く統合テスト | ネットワーク依存・レート消費・CI 不安定 |

**判断基準**: 「テストを書くコストが、得られる品質保証を下回るとき、そのテストは書かない。」

## 6. 突っ込み質問への回答

### Q: テストカバレッジはどのくらい？

「カバレッジの数値は追っていません。壊れたら困る箇所を重点的にカバーする方針で、スロット制限・上書き保護・エラーマッピング・CASCADE の4つを絶対に守るようにしています。」

### Q: なぜ ViewModel のテストを書いたの？

「UiState の遷移と Event の発行を検証するためです。例えばスロット満杯のときに SlotFull イベントが発行されるか、オフライン時に Markdown モードに切り替わるか、といった挙動が壊れると UX に直結するのでテストしています。」

### Q: Mock と実際の挙動がずれない？

「ずれる可能性はあります。Mock の設定が実際の API の挙動と乖離すると、テストは通るが本番で落ちる、という状況になり得ます。そのため、『何を検証しているか』をテスト名で明確にし、Mock の前提をコメントで残すようにしています。DAO は実際の SQLite で動かしているので、SQL まわりは実動に近い検証ができています。」

### Q: テスト駆動開発（TDD）をしている？

「全面的にはしていません。ただし、AI に実装を依頼するときは『先にテストを書いてから AI に実装させる』というテスト駆動AI開発をやっています。テストが通るまで修正を繰り返すので、AI 生成コードでも信頼性を担保できます。」

### Q: テストが増えすぎてメンテナンスが大変では？

「書くテストを絞っているので、現状はそこまで負荷はありません。定数や『1回呼ぶ』だけのテストは書かない方針で、壊れたら困る箇所に集中しています。テストが落ちたときに『何が壊れているか』が即座にわかるテスト名にしているので、修正も追いやすいです。」
