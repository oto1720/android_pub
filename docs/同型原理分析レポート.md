# 同型原理（Isomorphism Principle）に基づくプロダクト分析レポート

> **同型原理**: 「同じことは同じように扱う」— 一貫性のあるコードは可読性を高め、異質な部分（バグの温床）を目立たせる。

本レポートは、AI支援開発で見落とされがちな同型原理の観点から、本プロダクトの一貫性を分析した結果です。コードレビューでは diff のみが表示されるため、本来注視すべき「他箇所との整合性」が表示外になるリスクを補完する目的で作成しました。

---

## 2. 高優先度：アーキテクチャの不整合

### 2.1 Domain が UI に依存している（層の逆転）

**場所**: `domain/usecase/GetDoneArticlesUseCase.kt`

```kotlin
import com.example.oto1720.dojo2026.ui.done.DoneItem  // ← domain が ui を参照
```

- **問題**: Clean Architecture では domain は ui に依存してはならない。`GetDoneArticlesUseCase` が UI パッケージの `DoneItem` を返している。
- **他との比較**: `ObserveTsundokuArticlesUseCase` は `ArticleEntity`（data層）を返す。`GetDoneArticlesUseCase` だけが UI モデルを返しており、**同型原理に反する**。
- **推測**: Done 画面用の結合モデルを domain に置くべきところを、UI に置いた結果、UseCase が UI を参照する形になった可能性。

**推奨対応**:
- `DoneItem` 相当を `domain/model/DoneArticle.kt` などに移動
- `GetDoneArticlesUseCase` は `Flow<List<DoneArticle>>` を返す
- UI で `DoneArticle` → `DoneItem`（表示用）にマッピング

---

### 2.2 DoneDetailViewModel が Repository を直接使用

**場所**: `ui/done/DoneDetailViewModel.kt`

```kotlin
@HiltViewModel
class DoneDetailViewModel @Inject constructor(
    ...
    private val articleRepository: ArticleRepository,   // ← 直接注入
    private val digestRepository: DigestRepository,     // ← 直接注入
) : ViewModel() {
    private fun loadDetail() {
        val article = articleRepository.getArticleById(articleId)
        val digest = digestRepository.getByArticleId(articleId)
        ...
    }
}
```

- **問題**: 他 ViewModel（Portal, Digest, Tsundoku, Done）はすべて **UseCase 経由** でデータ取得している。`DoneDetailViewModel` だけが Repository を直接呼んでいる。
- **同型原理**: 「記事詳細を取得する」という同じ責務が、他画面では UseCase、DoneDetail では Repository と**形が異なる**。

**推奨対応**:
- `GetDoneDetailUseCase` を追加し、`articleRepository` + `digestRepository` の結合ロジックを移す
- `DoneDetailViewModel` は `GetDoneDetailUseCase` を注入し、他 ViewModel と同様の形にする

---

### 2.3 UI が data 層の Entity を直接利用

**場所**: `PortalUiState.kt`, `DigestUiState.kt`, `TsundokuUiState.kt`

```kotlin
// PortalUiState.kt
data class Success(val articles: List<ArticleEntity>, ...)  // ← data 層の Entity

// DigestUiState.kt
data class ReadMode(val article: ArticleEntity) : DigestUiState

// TsundokuUiState.kt
data class Success(val articles: List<ArticleEntity>)
```

- **問題**: UI が `ArticleEntity`（Room の DB モデル）を直接保持している。一方で `DoneUiState` は `DoneItem`（UI 用モデル）を使用している。
- **同型原理**: 「記事を表示する」という同じ責務で、**Portal/Tsundoku/Digest は Entity 直渡し**、**Done は UI モデル** と形が分かれている。
- **リスク**: DB スキーマ変更時に UI まで影響が及ぶ。表示専用のフィールド追加も Entity を汚すことになる。

**推奨対応**:
- `ArticleUiModel` や `ArticleDisplay` などを domain または ui に定義
- Repository/UseCase で `ArticleEntity` → `ArticleUiModel` にマッピング
- 全画面で UI モデルを統一して使用

## 3. 中優先度：状態管理・エラーハンドリングの不統一

### 3.1 状態の持ち方の違い

| ViewModel | uiState の生成方式 | イベント |
|-----------|-------------------|----------|
| PortalViewModel | `MutableStateFlow` + `asStateFlow()` | `SharedFlow<PortalEvent>` |
| DigestViewModel | `MutableStateFlow` + `asStateFlow()` | `SharedFlow<DigestEvent>` |
| DoneDetailViewModel | `MutableStateFlow` + `asStateFlow()` | なし |
| **DoneViewModel** | **`Flow.map().stateIn()`** | なし |
| **TsundokuViewModel** | **`Flow.map().stateIn()`** | なし |

- **問題**: 単純な Flow 変換（Done, Tsundoku）と、MutableStateFlow による手動更新（Portal, Digest, DoneDetail）が混在。
- **評価**: 実装としてはどちらも妥当だが、「同じ種類の画面（一覧系）でも形が違う」という点で同型原理にやや反する。将来的な拡張時にどちらのパターンに寄せるか迷いが生じる。

---

### 3.2 UseCase の戻り値の不統一

| UseCase | 戻り値 | 失敗時 |
|---------|--------|--------|
| `GetArticleUseCase` | `ArticleEntity?` | null |
| `GetTrendArticlesUseCase` | `Result<List<ArticleEntity>>` | Result.failure |
| `GenerateQuestionUseCase` | `Result<String>` | Result.failure |
| `GenerateFeedbackUseCase` | `Result<String>` | Result.failure |

- **問題**: 同種の「取得系」UseCase で `T?` と `Result<T>` が混在。呼び出し側のエラーハンドリングが画面ごとに異なる。
- **例**: `DigestViewModel` は `getOrDefault(PLACEHOLDER_*)` で AI エラーを握りつぶしている。ユーザーにはエラーが伝わらない。

---

### 3.3 UiState の Error の有無の不統一

| UiState | Error 定義 |
|---------|------------|
| PortalUiState | ✅ `Error(val message: String)` |
| DigestUiState | ✅ `Error(val message: String)` |
| DoneDetailUiState | ✅ `Error(val message: String)` |
| **DoneUiState** | ❌ なし |
| **TsundokuUiState** | ❌ なし |

- **問題**: 一覧系画面（Done, Tsundoku）は Error がない。取得失敗時の扱いが画面ごとに異なる。
- **背景**: Done/Tsundoku は Room の Flow をそのまま使っており、ローカル DB なので「失敗」が想定されにくい設計かもしれない。ただし、将来的にネットワーク連携などが入ると不整合が顕在化する。

## 4. 低優先度：その他の観点

### 4.1 DigestViewModel のエラー握りつぶし

```kotlin
// DigestViewModel.kt 69-70, 85-86 行
val question = generateQuestionUseCase(articleId).getOrDefault(PLACEHOLDER_QUESTION)
val feedback = generateFeedbackUseCase(...).getOrDefault(PLACEHOLDER_FEEDBACK)
```

- AI API 失敗時にプレースホルダーを表示し、ユーザーにエラーを通知しない。
- 仕様として意図的であれば問題ないが、他画面（Portal など）では `PortalEvent.ShowOfflineMessage` で通知している。**エラー時の振る舞いが画面間で不統一**。

---

## 5. 同型原理の観点で「良い点」

- **命名規則**: ViewModel, Repository, UseCase の命名は一貫している
- **DI**: Hilt の使い方（@HiltViewModel, @Inject）は統一されている
- **StateFlow + collectAsState**: UI 層の状態購読は統一されている
- **ディレクトリ構造**: data / domain / ui のレイヤー分離は明確

---

## 6. 推奨アクション（優先度順）

1. **Domain の UI 依存解消**: `GetDoneArticlesUseCase` から `DoneItem` 参照を外し、domain モデルを導入
2. **DoneDetailViewModel の UseCase 化**: `GetDoneDetailUseCase` を追加し、Repository 直接参照をやめる
3. **Entity の UI 露出をやめる**: `ArticleUiModel` 等を導入し、全画面で UI モデルに統一
4. **UseCase 戻り値の統一**: 取得系を `Result<T>` に統一するか、設計方針をドキュメント化
5. **UiState の Error 統一**: 一覧系画面にも Error を定義するか、エラーは SharedFlow で扱うか方針を決める
6. **状態管理パターンの方針化**: `MutableStateFlow` vs `Flow.stateIn()` の使い分けをルール化

---

## 7. コードレビュー時のチェックリスト（同型原理）

diff のみを見るコードレビューでは、以下を**diff 外**で確認することを推奨します：

- [ ] 同じ責務の処理が、他画面では UseCase / 今回の変更では Repository 直参照になっていないか
- [ ] 新規 UiState が、他画面の Error の有無と整合しているか
- [ ] 新規 UseCase の戻り値が、既存の `Result` / `T?` の方針と一致しているか
- [ ] domain 層が ui パッケージを import していないか
- [ ] UI が data 層の Entity を直接参照していないか（新規画面の場合）
