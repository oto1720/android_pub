### DigestRepository.kt / DigestRepositoryImpl.kt

#### 変更の意図
`GenerateFeedbackUseCase` がフィードバックを保存する口として `saveFeedback(articleId, feedback)` を追加。実装は `saveQuestion` と完全に対称。

```kotlin
// DigestRepository.kt
/**
 * フィードバックを保存する。既存の [DigestEntity] があれば [DigestEntity.aiFeedback] のみ更新し、
 * なければ新規作成する。
 */
suspend fun saveFeedback(articleId: String, feedback: String)

// DigestRepositoryImpl.kt
override suspend fun saveFeedback(articleId: String, feedback: String) {
    val existing = digestDao.getByArticleId(articleId)
    if (existing != null) {
        digestDao.update(existing.copy(aiFeedback = feedback))
    } else {
        val savedAt = SimpleDateFormat(ISO_8601_FORMAT, Locale.US)
            .apply { timeZone = TimeZone.getTimeZone("UTC") }
            .format(Date())
        digestDao.insert(DigestEntity(articleId = articleId, aiFeedback = feedback, savedAt = savedAt))
    }
}
```

#### 良い点
- `saveQuestion` と同一パターンで、`copy(aiFeedback = feedback)` により他フィールド（`aiQuestion` / `userMemo`）を保持したまま更新される
- `ISO_8601_FORMAT` 定数を再利用しておりフォーマットの統一が保たれている

### GenerateFeedbackUseCase.kt

#### 変更の意図
フィードバック生成に特化したUseCase。`AiRepository.generateFeedback(question, userMemo)` を呼び、成功時のみ `DigestRepository.saveFeedback` で保存する。

```kotlin
suspend operator fun invoke(
    articleId: String,
    question: String,
    userMemo: String,
): Result<String> {
    return aiRepository.generateFeedback(question, userMemo).onSuccess { feedback ->
        digestRepository.saveFeedback(articleId, feedback)
    }
}
```

#### 良い点
- フィードバックはユーザーのメモに依存するため、`GenerateQuestionUseCase` のようなキャッシュ機構を持たない設計が適切
- `.onSuccess { }` でAPIエラー時の `saveFeedback` 呼び出しを自動防止
- `ArticleRepository` への依存がなく、必要最小限の依存関係
