### build.gradle.kts

#### 変更の意図
`local.properties` から `GEMINI_API_KEY` を読み取り、`BuildConfig` フィールドとして埋め込む。`GEMINI_BASE_URL` も同様に定数として管理する。

```kotlin
val localProperties = Properties()
val localPropertiesFile = rootProject.file("local.properties")
if (localPropertiesFile.exists()) {
    localProperties.load(localPropertiesFile.inputStream())
}
val geminiApiKey: String = localProperties.getProperty("GEMINI_API_KEY", "")
```

#### 良い点
- `local.properties` が存在しない場合のフォールバック（`""` デフォルト）でビルドが落ちない
- `GEMINI_BASE_URL` を BuildConfig で一元管理しており、URL のハードコードを避けている
- `import java.util.Properties` をファイル先頭に追加しており Kotlin DSL の制約を正しく対処している

### NetworkModule.kt

#### 変更の意図
`@Named("gemini")` qualifier で Gemini 専用の OkHttpClient・Retrofit を提供。API キーインターセプターを OkHttpClient に追加することでリクエストごとに `?key=...` を自動付与する。

```kotlin
@Named("gemini")
@Provides
@Singleton
fun provideGeminiOkHttpClient(loggingInterceptor: HttpLoggingInterceptor): OkHttpClient =
    OkHttpClient.Builder()
        ...
        .addInterceptor { chain ->
            val original = chain.request()
            val url = original.url.newBuilder()
                .addQueryParameter("key", BuildConfig.GEMINI_API_KEY)
                .build()
            chain.proceed(original.newBuilder().url(url).build())
        }
        .build()
```

#### 良い点
- `loggingInterceptor` を Gemini 用 OkHttpClient でも共有しており、ログ設定が統一される
- qualifier の使い方が Hilt の推奨パターンに準拠
