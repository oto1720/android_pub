## 🔍 詳細レビュー

### navigation/Screen.kt

#### 変更の意図
ルートを文字列リテラルで各所に散らばらせず、sealed class で一元管理する。

#### 指摘事項

**[🟡 M-1] 引数付き Screen インスタンスの `route` プロパティが misleading** (`Screen.kt:8-22`)

```kotlin
// ❌ 現状: Digest("someId").route == "digest/{articleId}" (テンプレート文字列が返る)
data class Digest(val articleId: String) : Screen("digest/{articleId}") {
    companion object {
        const val ROUTE = "digest/{articleId}"
        ...
    }
}
```

```kotlin
// ✅ 案1: route を解決済みの文字列にオーバーライド（インスタンスから直接 navigate できる）
data class Digest(val articleId: String) : Screen("digest/$articleId") {
    companion object {
        const val ROUTE = "digest/{articleId}"
        const val ARG = "articleId"
        fun createRoute(articleId: String) = "digest/$articleId"
    }
}

// ✅ 案2: 引数なし screens のみ sealed class にして、引数ありは companion object だけにする
//   （シンプルさ重視で、AppNavHost 側の使い方は変わらない）
```

> `Digest("id").route` と `Digest.createRoute("id")` が別の値を返すため、将来インスタンスの `route` を使うコードが書かれると混乱の原因になりうる。現状の `AppNavHost` では `companion object` の定数・関数を正しく使っているため動作上の問題はない。
