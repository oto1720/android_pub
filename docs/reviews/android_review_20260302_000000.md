## Major Issues

### [M-1] compileSdk と targetSdk の乖離

**ファイル**: `app/build.gradle.kts:11-18`
**カテゴリ**: Build Configuration

```kotlin
// 現在のコード
android {
    compileSdk = 36   // API 36
    defaultConfig {
        targetSdk = 35  // API 35 ← compileSdk と 1 世代ズレている
    }
}
```

```kotlin
// 推奨
android {
    compileSdk = 36
    defaultConfig {
        targetSdk = 36  // compileSdk に揃える
    }
}
```

**なぜ問題か**: `compileSdk` は「どの API レベルでコンパイルするか」、`targetSdk` は「このアプリはどの API レベルの動作を期待するか」を示す。
通常この 2 つは一致させる。乖離があると、compileSdk=36 で導入された新しい制約や動作変更を targetSdk=35 で意図的に回避しているように見え、
コードレビューア・Google Play 審査者の双方にとって意図が不明確になる。
意図的に古い targetSdk を指定するケースは段階的移行時に限られ、その場合はコメントで理由を明記すべき。

**参考**: [Android API levels - target SDK](https://developer.android.com/tools/releases/platforms)

### [M-2] Compose BOM が古い

**ファイル**: `gradle/libs.versions.toml:12`
**カテゴリ**: Dependency Version

```toml
# 現在
composeBom = "2024.09.00"
```

```toml
# 推奨 (2026-03-02 時点の最新安定版)
composeBom = "2025.05.00"  # 実際の最新版を Compose BOM リリースノートで確認すること
```

**なぜ問題か**: Compose BOM `2024.09.00` は約 1.5 年前のリリースであり、
Kotlin 2.0.21 との相互運用や Compose Compiler の最適化において既知の改善が積み重なっている。
特に `kotlin-compose` プラグイン (`2.0.21`) と BOM 内の Compose Compiler バージョンの整合性が
古い BOM では保証されない場合がある。BOM を最新安定版に更新することで、
Compose Compiler との相性問題を回避し、パフォーマンス改善の恩恵も受けられる。

**参考**: [Compose BOM to library version mapping](https://developer.android.com/jetpack/compose/bom/bom-mapping)

## Minor Issues

### [Mi-1] AppModule が空の object として定義されている

**ファイル**: `app/src/main/java/com/example/oto1720/dojo2026/di/AppModule.kt:7-9`
**カテゴリ**: Architecture / Hilt

```kotlin
// 現在のコード
@Module
@InstallIn(SingletonComponent::class)
object AppModule
```

```kotlin
// 将来のバインディング追加を見据えた形 (プロバイダーが 1 件でも追加される段階でこの形にする)
@Module
@InstallIn(SingletonComponent::class)
object AppModule {

    @Provides
    @Singleton
    fun provideExampleRepository(
        // 依存関係
    ): ExampleRepository = ExampleRepositoryImpl()
}
```

**なぜ問題か**: 現時点では空の Module はコンパイルエラーにならず、Hilt は問題なく動作する。
ただし、空の `@Module` は KSP がコード生成する対象として認識されるため、ビルド時間に僅かながら影響を与える。
より重要な点として、`@Provides` も `@Binds` も存在しない Module を宣言し続けることは、
「将来どんな依存関係をここに追加するか」という設計意図がコードから読み取れず、チームメンバーへの混乱を招く。
**issue #3 がセットアップ専用 issue であれば現状は許容できる** が、
次の issue で最初のプロバイダーを追加した段階でボディを記述することを推奨する。

### [G-1] KSP を正しく選択している

**ファイル**: `gradle/libs.versions.toml:4` / `app/build.gradle.kts:55`

```toml
ksp = "2.0.21-1.0.27"
```

```kotlin
ksp(libs.hilt.compiler)
```

Kotlin 2.0.x 環境において kapt ではなく KSP を採用した判断は正しい。
kapt は Kotlin 2.0 以降で deprecated 扱いとなっており、
KSP はビルド時間が kapt 比で大幅に短縮される (プロジェクト規模に依存するが 2〜3 倍高速なケースも報告されている)。
KSP バージョン `2.0.21-1.0.27` は Kotlin `2.0.21` と接頭辞が一致しており、バージョン整合性も取れている。

### [G-2] Version Catalog (libs.versions.toml) を活用している

Gradle Version Catalog によりバージョン定義が一元管理されている。
