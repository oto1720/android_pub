### PortalViewModel.kt

#### 変更の意図
① オフライン遷移を `isOnline.drop(1).collect` で監視、② API 失敗時に Room キャッシュへフォールバックし、両方で `ShowOfflineMessage` イベントを emit する。

#### 良い点

**[🟢 G-2] `drop(1)` でアプリ起動時の初期値をスキップ** (`PortalViewModel.kt:72`)
```kotlin
isOnline
    .drop(1) // 初期値（true）をスキップし、変化のみに反応する
    .collect { online ->
        if (!online) {
            _event.emit(PortalEvent.ShowOfflineMessage)
        }
    }
```
アプリ起動直後に不要な Snackbar が表示されない。

**[🟢 G-3] `runCatching` でキャッシュ取得失敗を安全に処理** (`PortalViewModel.kt:127`)
```kotlin
val cached = runCatching { observePortalArticlesUseCase().first() }
    .getOrDefault(emptyList())
```
キャッシュ自体の取得が失敗しても二重エラーにならず、空リストとして扱われて `PortalUiState.Error` へ graceful に遷移する。

**[🟢 G-4] キャッシュ有無で Success/Error を分岐**
```kotlin
if (cached.isNotEmpty()) {
    _uiState.value = PortalUiState.Success(cached, availableTags)
} else {
    _uiState.value = PortalUiState.Error(error.message ?: "Unknown error")
}
_event.emit(PortalEvent.ShowOfflineMessage)
```
キャッシュがある場合は Success として記事を表示し続け、キャッシュがない場合のみ Error を表示する。Snackbar は両ケースで発行されるため、ユーザーがオフライン状態を認識できる。

### PortalViewModelTest.kt

#### 変更の意図

#### 良い点

**[🟢 G-5] コレクターを先に購読してからリトライで失敗させる正しいパターン**
```kotlin
createViewModel()
runCurrent()
val collectJob = launch { viewModel.event.collect { receivedEvents.add(it) } }
```

```kotlin
val onlineFlow = MutableStateFlow(true)
every { networkMonitor.isOnline } returns onlineFlow
// ...
```
