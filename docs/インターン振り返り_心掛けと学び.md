# CA Tech Dojo インターン振り返り — 心掛けたことと学んだこと

## CA Tech Dojo とは

CA Tech Dojo は、株式会社 CyberAgent が開催している学生向けの育成型インターンシップです。

今回は **Android 開発**をテーマに、1週間で Android アプリ開発の基礎スキルを身につけるプログラムでした。全体のゴールは「**1人でアプリを開発しリリースできるようになる**」ことです。

- [CA Tech Dojo 2026 詳細](https://www.cyberagent.co.jp/careers/students/event/detail/id=32667)

---

## インターンでの目標

開始前に設定した目標は次の通りです。

> **Jetpack Compose を使って Android アプリ開発の実践力を身につけ、設計・実装ができるようになる**

これまでは Flutter での実践的な開発が主で、Android アプリの開発経験はほとんどなく、Kotlin もほぼ初めてでした。そのため、

- Jetpack Compose を用いた UI 実装
- Android アプリの基本的な設計
- API を利用したデータ取得
- 状態管理

といった Android 開発の基本的な流れを一通り経験することを目標としました。

---

## インターンでの課題と開発したもの

お題は「**コンテンツポータルアプリ**」。機能要件・任意要件・非機能要件が与えられ、1週間という期間の中でどこまで実装できるかが問われる形式でした。

### 作ったアプリ：Tech-Digest

**技術記事の「読んだつもり」をなくす理解完了アプリ**を開発しました。

技術記事は「読んだだけ」で終わってしまい、実際には理解できていないことが多いと感じていたため、**記事を理解するところまでサポートするアプリ**として設計しました。

| 画面 | 主な機能 |
|------|----------|
| **ポータル画面** | Qiita API からトレンド記事を取得・表示、タグ別絞り込み、「スロットに追加」ボタン（最大5枠） |
| **積読リスト画面** | 最大5枠、AI要約ボタン（Gemini API）、記事本文を読んで理解度チェックへ |
| **消化フロー** | AI が問いを生成 → 1行メモ入力 → AI がフィードバックと理解度評価（十分/不十分）→ 十分なら「消化する」で完了リストへ |
| **完了リスト画面** | 消化済み記事・メモ・AI フィードバックの確認 |

要件が多く、AI を活用することを前提とした開発環境だと感じました。実際の開発でも AI 活用が一般的になってきており、**AI をどれだけ使いこなせるか**が重要になっていると感じ、その観点も意識して臨みました。

## インターンで心掛けたこと

今回のインターンでは主に次の 2 つを意識して取り組みました。

### 1. 自分のやりたいことの軸をぶらさないこと

インターンでは、Jetpack Compose で UI を作り込みたい人、プロダクトの完成度を高めたい人、新しい技術を試したい人など、やりたいことが人によって異なります。

その中で自分は**優先順位を決めて開発を進めること**を意識しました。今回は **設計を第一に考える**という方針で進め、UI などは二の次にして、

- コードの構成
- アーキテクチャ
- データの流れ

といった**設計としてどこまで整理できるか**を重視しました。

一人での開発・Kotlin 初経験・Android 初経験という状況だったため、**公式のベストプラクティスを参考に設計を進める**ことにしました。

インターン中は他の参加者の実装を見る機会も多く、「これもやってみたい」と思うこともありました。ただ、周りに影響されすぎると本来自分が学びたいことや目標が曖昧になると感じたため、**参考になる部分は取り入れつつ、自分の軸はぶらさない**ことを意識しました。特に設計では、**どれだけ一貫性を保てるか**を強く意識して実装を進めました。

### 2. AI 前提開発で意識したこと

昨今では AI の急速な進化により、コードをすべて自分で書く場面は以前より少なくなってきています。そのような環境の中で、エンジニアとして**どのように Value を出していくのか**を常に考えて開発を行いました。

今回特に意識したことを 3 つ紹介します。

#### AI の出力したものの品質をどう担保するか

AI は高速にコードを生成してくれますが、その出力が必ずしも最適な実装とは限りません。実際には、

- 設計と合わない実装になる
- 不要に複雑なコードになる
- 一貫性のないコードになる

といったケースもありました。

そのため AI が生成したコードをそのまま使うのではなく、

- なぜこの実装になっているのか
- アプリ全体の設計と矛盾していないか
- コードの一貫性が取れているか

といった点を確認しながら取り入れるようにしました。**AI を使う場合でも、最終的に品質を担保するのは開発者自身**であるという意識が重要だと感じました。

#### AI の役割を明確にする

AI は非常に強力で、コード生成やリファクタリングなど多くの作業を高速に行ってくれます。しかし実際に開発を進めてみると、

> **AI に任せすぎると、かえって開発が進まなくなる**

という場面もありました。例えば、

- 設計意図とズレたコードが生成される
- その場しのぎの実装が増える
- コードの一貫性が崩れる

といった問題が起こることがあります。その結果、**AI のコードを修正する時間**の方が長くなるという状況もありました。

そこで今回の開発では、**AI に任せる部分と、自分が責任を持つ部分を明確に分ける**ことを意識しました。設計やアーキテクチャの判断は自分が行い、AI には実装の具体化や定型処理を任せるようにしました。

#### AI を使った開発フローを作る

AI の出力を信頼しつつも、品質を担保するために次のような開発フローを用意しました。

| 意識 | 具体例 |
|------|--------|
| **テスト駆動 AI 開発** | 先にテストを書いてから AI に実装依頼。テストが通るまで修正することで、AI 生成コードでも信頼性を担保 |
| **仕様駆動開発（SDD）** | design.md や technical-design.md を AI に渡して実装。設計意図がコードに反映されるようにした |
| **Skills の活用** | code-review、pr-review、isomorphism-check などで、AI の出力を人間の観点で検証 |
| **何を検証しているか明確に** | テスト名で「壊れたら困る箇所」を明示。Mock の前提をコメントで残し、AI に任せても意図が伝わるようにした |

## 学んだこと

### 技術面

| 学び | 内容 |
|------|------|
| **Clean Architecture の層分離** | ドメインを UI から切り離すとテストが書きやすく、変更に強い。UseCase を独立させることで、Repository の実装を変えても影響範囲を限定できる |
| **StateFlow と SharedFlow の使い分け** | StateFlow は「今の状態」を保持、SharedFlow は「起きたこと」を流す（一度きりのイベント）。状態とイベントを区別することで、UI の設計が明確になる |
| **AI を UX に組み込む** | プロンプトで出力形式を厳密に指定（例：`JUDGMENT: OK/NG`）。キャッシュで API 呼び出しを減らし、UX を高速に保つ |
| **テストの考え方** | 「壊れたら気づく仕組み」が目的。壊れたら困る箇所（スロット制限、上書き保護、エラーマッピングなど）を重点的にテストし、定数や「1回呼ぶだけ」のテストは書かない |

### プロセス面

| 学び | 内容 |
|------|------|
| **同型原理** | 「同じことは同じように扱う」。エラー処理を AppError に一元化するなど、同じ責務の処理が画面ごとに形が違わないようにする |
| **設計の言語化** | 選定理由やトレードオフを technical-design.md に記載。AI にも人間にも、意図が伝わるようにする |
| **AI との協働** | AI に任せる部分と、人間が検証する部分を分ける。Skills やテストで、AI の出力をチェックする仕組みを作る |

---

## まとめ

1週間という短期間で、要件の多いアプリを設計・実装する経験を通じて、**Jetpack Compose と Clean Architecture の実践**、**AI を前提とした開発プロセス**の両方を学びました。

特に「自分の軸をぶらさない」「AI を道具として使いこなす」という意識を持ち続けたことで、単に動くアプリではなく、**設計意図が明確で、壊れても気づける仕組みを持つ**アプリを作れたと思います。
