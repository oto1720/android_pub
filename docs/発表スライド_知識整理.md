- Qiita・Zenn を読んでも実装時に思い出せない経験 → それを解決

### 押さえるポイント

| 用語 | 意味 |
|------|------|
| **読んだつもり** | 読んだ瞬間はわかった気になるが、知識が定着していない状態。実装時に思い出せない。 |
| **Tech-Digest** | 技術記事の積読管理＆AI消化アプリ。Android Dojo 2026 インターンプロダクト。 |

### 突っ込み対策

- **「具体的にどう解決するの？」** → スライド3・4で説明。強制1行メモ（AI壁打ち）と積読5件制限で「読んだだけでは消化にならない」「まず消化してから次へ」を強制する。

## スライド2｜課題提起（約50秒）

### 話す内容

3つの課題：
1. **読んだつもり問題** — 知識が定着していない
2. **心理的負債** — ブックマークが無限に増え、焦りだけが増える
3. **学習の迷子** — つまみ食いで体系的理解ができず、チュートリアル地獄

### 押さえるポイント

| 課題 | 解決策（スライド4で詳述） |
|------|---------------------------|
| 読んだつもり | 強制1行メモ（AI壁打ち）。AIが問いを出し、答えて初めて消化 |
| 心理的負債 | 積読スロット制限（最大5件）。追加できなくなる |
| 学習の迷子 | オフライン集中モード。1記事に集中させる |

### 突っ込み対策

- **「なぜ5件なの？」** — 少なすぎるとストレス、多すぎると負債が溜まる。5件は「今やるべきこと」に集中させるバランスの良い数として設定。
- **「ブックマークアプリと何が違う？」** — 読んだだけでは完了にならない。AIが理解度をチェックし、合格しないと消化できない点が決定的な違い。

## スライド3｜主要機能デモ（約60秒）

### 話す内容

4ステップの使い方：
1. **ポータル** — Qiitaトレンド取得、タグ絞り込み
2. **積読リスト** — 最大5枠、AI要約3行
3. **消化フロー** — AI問い → 回答 → 消化完了
4. **完了リスト** — メモとセットで保存、知識ベースとして蓄積

### 押さえるポイント

| 画面 | 主な機能 | 技術的には |
|------|----------|------------|
| ポータル | Qiita API、タグ絞り込み、積読追加 | Paging3, QiitaRemoteMediator, ArticleDao |
| 積読リスト | 最大5枠、AI要約 | GenerateSummaryUseCase, ArticleEntity.aiSummary |
| 消化フロー | AI問い、1行メモ、理解度評価 | GenerateQuestionUseCase, GenerateFeedbackUseCase, DigestEntity |
| 完了リスト | 消化済み一覧、メモ・フィードバック確認 | GetDoneArticlesUseCase, DigestEntity |

### 実機デモの流れ（推奨）

1. ポータルでタグ選択 → 記事を積読に追加
2. 積読リストでAI要約ボタン → 記事タップで消化画面へ
3. 読了宣言 → AI問い表示 → 1行メモ入力 → 送信 → フィードバック → 消化する
4. 完了リストでメモ・フィードバックを確認

### 突っ込み対策

- **「AI要約は何を渡している？」** — 記事のタイトルではなく**全文（body）**。Qiita API の body を Room に保存し、Gemini に渡している。
- **「問いは毎回変わる？」** — 記事ごとに内容に即した問いを動的生成。同じ記事なら DB にキャッシュしてあるので2回目以降は即表示。

## スライド4｜失敗ケースとUXの工夫（約30秒）

### 話す内容

- **意図的な制約** — 積読5件で追加不可、まず消化してから次へ
- **理解不十分で消化不可** — AIがNGなら適当な1行では通らない
- **オフライン対応** — キャッシュ表示で電波なしでも学習継続

### 押さえるポイント

| 工夫 | 実装 |
|------|------|
| 積読5件制限 | ArticleDao.updateStatusIfSlotAvailable。SQL でアトミックにチェック |
| 理解不十分で消化不可 | GenerateFeedbackUseCase が JUDGMENT: NG を返すと「消化する」ボタン無効 |
| オフライン | NetworkMonitor、Room キャッシュ、Snackbar 通知 |

### 突っ込み対策

- **「なぜ制約がUXを良くするの？」** — 選択肢が多すぎると迷い、負債が溜まる。5件に制限することで「今やるべきこと」が明確になり、消化したときの達成感が得られる。
- **「オフラインで記事は読める？」** — body を Room に保存しているので、Markdown 表示で読める。WebView は URL が必要なためオフラインでは不可。

## スライド5｜技術スタック（約40秒）

### 話す内容

- Kotlin, Jetpack Compose（モダン構成）
- Clean Architecture + MVVM（3層分離、一方向依存）
- Room, Paging3, Hilt, Retrofit, Gemini API
- 選定理由は technical-design.md に記載

### 押さえるポイント

| 技術 | 役割 | 選定理由（要約） |
|------|------|------------------|
| **Clean Architecture** | UI・Domain・Data の3層 | テスタビリティ、責務の明確さ。層は一方向にしか依存しない |
| **MVVM** | ViewModel が状態を保持 | 学習コストが低く、UiState sealed class で MVI 的な単方向性を確保 |
| **Room** | ローカルDB | 記事・消化記録のリレーション。DataStore は Key-Value 向き |
| **Paging3** | 無限スクロール | オフラインキャッシュと再同期を RemoteMediator で両立 |
| **Hilt** | DI | コンストラクタ注入、テストで Mock に差し替え可能 |
| **Retrofit** | ネットワーク | デファクトスタンダード、Hilt との相性 |
| **Gemini API** | AI | REST で実装。SDK ではなく Retrofit 経由で、将来の切り替えを考慮 |

### 層の依存関係（図の説明用）

```
UI層（Compose, ViewModel）
    ↓ 依存
Domain層（UseCase, Model, Repository Interface）
    ↑ 実装
Data層（Repository Impl, API, Room）
```

- Domain は UI や Data に依存しない
- Data が Domain の interface を実装する

### 突っ込み対策

- **「なぜ MVI にしなかった？」** — 学習コストが高い。UiState sealed class で十分な単方向性を確保し、実装コストを抑えた。
- **「Gemini を REST にした理由？」** — 既存の NetworkModule に統合できる。実装が透明で、OpenAI/Claude への切り替えも容易。

## スライド6｜データフローとオフライン対応（約30秒）

### 話す内容

- **SSOT（Single Source of Truth）** — API → Room に保存、UI は Room だけを見る
- **オフライン** — 同じ UI が動く理由。NetworkMonitor で監視、Snackbar で通知

### 押さえるポイント

| 用語 | 意味 |
|------|------|
| **SSOT** | データの真実の源を1つにすること。このアプリでは Room が真実の源。API は Room を更新するだけ。 |
| **データの流れ** | Qiita API → RemoteMediator が Room に upsert → PagingSource が Room から読み取り → UI に表示 |
| **NetworkMonitor** | ConnectivityManager で接続状態を監視。Flow&lt;Boolean&gt; で isOnline を流す |
| **オフライン時** | Room のキャッシュを表示。Snackbar「オフラインです」を表示。オンライン復帰で RemoteMediator が再フェッチ |

### 突っ込み対策

- **「UI が API を直接見ないメリットは？」** — オフライン時も同じコードパスで動く。キャッシュがあれば表示される。オンライン/オフラインで分岐しない。
- **「NetworkMonitor はどう実装？」** — ConnectivityManager.NetworkCallback を callbackFlow でラップし、Flow&lt;Boolean&gt; として公開。distinctUntilChanged で変化時のみ emit。

## スライド8｜品質へのこだわり（約30秒）

### 話す内容

1. **同型原理** — 同じことは同じように。AppError でエラー処理を一元化
2. **テスト** — 壊れたら困るところを重点的に（スロット制限、エラーマッピング等）
3. **設計の言語化** — technical-design.md に選定理由とトレードオフを記載

### 押さえるポイント

| 用語 | 意味 |
|------|------|
| **同型原理** | 「同じことは同じように扱う」。同じ責務の処理が画面ごとに形が違うとバグの温床になる。 |
| **AppError** | sealed class。Unauthorized, Forbidden, RateLimitExceeded, NetworkError, AiError 等。Repository で toAppError() でマッピング。 |
| **重点テスト** | スロット制限（ArticleDaoTest）、上書き保護（ArticleRepositoryImplTest）、429→RateLimitExceeded、外部キー CASCADE |
| **technical-design.md** | パッケージ構成、クラス図、MVVM vs MVI、Room vs DataStore、Retrofit vs Ktor、Gemini REST vs SDK 等のトレードオフを記載 |

### 突っ込み対策

- **「同型原理の具体例は？」** — エラー処理を各画面でバラバラに書かず、AppError に一元化。他には、他 ViewModel は UseCase 経由なのに DoneDetail だけ Repository 直参照、という不整合が同型原理レポートで指摘されている。
- **「なぜ E2E テストを書かない？」** — コストが高く壊れやすい。スロット制限やエラーマッピングなど、壊れたら UX が崩壊する箇所を単体・統合テストで重点的にカバーする方針。

## スライド9｜AI活用（アプリ機能）（約30秒）

### 話す内容

1. **AI要約** — 記事全文を3行に。Room にキャッシュ、重複 API 呼び出しを防ぐ
2. **AI問い生成** — 記事ごとに内容に即した問い。汎用問いではないのでごまかせない
3. **Gemini REST** — SDK ではなく REST。OpenAI/Claude への切り替えを考慮

### 押さえるポイント

| 機能 | 入力 | 出力 | キャッシュ |
|------|------|------|------------|
| AI要約 | 記事全文（body） | 3行の要約 | ArticleEntity.aiSummary |
| AI問い | 記事全文 | 1つの問い | DigestEntity.aiQuestion |
| AIフィードバック | 問い+メモ+記事全文 | フィードバック+理解度OK/NG | DigestEntity.aiFeedback |

- **理解度評価** — Gemini が `JUDGMENT: OK` または `JUDGMENT: NG` を返す。NG なら「消化する」ボタン無効
- **REST の理由** — Retrofit に統合できる。実装が透明。SDK は内部が隠蔽され、切り替えが難しい

### 突っ込み対策

- **「記事全文を渡すとトークン数が多くない？」** — 長い場合は 10,000 文字で切り詰め。要約・問い・フィードバックは Room にキャッシュするので、同じ記事で何度も API を叩かない。
- **「理解度評価はどうパース？」** — レスポンスの `JUDGMENT: OK` / `JUDGMENT: NG` を正規表現や行パースで取得。パース失敗時は安全側で「理解十分」とみなす。

## スライド10｜AI活用（開発プロセス）（約30秒）

### 話す内容

1. **Claude Skills** — コードレビュー、PRレビューを AI に自動化。観点を統一
2. **テスト駆動AI開発** — 先にテストを書いてから AI に実装依頼。テストが通るまで修正
3. **SDD（仕様駆動開発）** — design.md を AI に渡して実装。設計意図がコードに反映
4. **Figma MCP** — デザインから Compose コード生成時のルールと連携

### 押さえるポイント

| 用語 | 意味 |
|------|------|
| **Claude Skills** | `.claude/skills/` に定義。code-review, pr-review, code-learner, feature-learner, isomorphism-check, isomorphism-review, android-test-generator 等 |
| **テスト駆動AI開発** | テストを先に書く → AI に実装依頼 → テストが通るまで修正。AI 生成コードでも信頼性を担保 |
| **SDD** | design.md（画面フロー、機能設計、DB スキーマ、AI フロー）を AI に渡す。technical-design.md も同様 |
| **Figma MCP** | Figma のデザインから UI コードを生成する際の MCP。compose-ui.mdc がルールファイルと連携 |

### Skills 一覧（質疑用）

| スキル | 用途 |
|--------|------|
| code-review | コードレビュー、指摘事項の Markdown 出力 |
| pr-review | PR・コミット差分のレビュー、マージ可否 |
| code-learner | 既存コードの学習用 MD 生成 |
| feature-learner | 新機能の学習 MD 生成 |
| isomorphism-check | 同型原理に基づく一貫性チェック |
| isomorphism-review | 7設計原則に基づくパターン一貫性レビュー |
| android-test-generator | テストコード生成 |

### 突っ込み対策

- **「Skills はどう作った？」** — SKILL.md という形式で、トリガーとなる文言と実行するタスクを定義。Cursor 上で Claude がそれを読み取り、特定のタスクを実行する。
- **「design.md を渡すとどうなる？」** — AI が画面フローや DB スキーマを理解した上で実装するため、設計の意図がコードにズレなく反映される。

## スライド11｜学んだこと・今後の展望（約30秒）

### 話す内容

**学んだこと**
1. Clean Architecture の層分離 — ドメインを UI から切り離すとテストが書きやすく、変更に強い
2. StateFlow と SharedFlow の使い分け — 状態 vs イベントの区別
3. AI を UX に組み込む — プロンプト設計とキャッシュ戦略がセットで重要

**今後の展望**
- WorkManager でバックグラウンド同期。アプリを開かなくてもトレンド更新

### 押さえるポイント

| 学び | 具体例 |
|------|--------|
| 層分離 | UseCase を独立させることで、ViewModel のテストが Mock で容易に。Repository の実装を変えても UseCase は影響を受けない |
| StateFlow vs SharedFlow | StateFlow：画面状態（一覧、選択中）。SharedFlow：一度きりイベント（Snackbar、ナビゲーション） |
| AI の UX | プロンプトで出力形式を厳密に指定（JUDGMENT: OK/NG）。キャッシュで API 呼び出しを減らし、UX を高速に |
| WorkManager | 定期的に Qiita API を呼び、Room を更新。アプリ起動時には最新のトレンドが既にキャッシュされている |

### 突っ込み対策

- **「StateFlow と SharedFlow の違いを一言で？」** — StateFlow は「今の状態」を保持する。SharedFlow は「起きたこと」を流す（再配信しない）。
- **「WorkManager の実装イメージは？」** — PeriodicWorkRequest で Qiita API を呼び、ArticleRepository の refreshPortalArticles 相当を実行。Room が更新されれば、Paging の Flow が自動で UI に反映される。
